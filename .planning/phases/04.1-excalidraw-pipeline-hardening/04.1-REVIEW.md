# Phase 04.1: Code Review Report

**Reviewed:** 2026-05-07T16:00:00Z
**Depth:** standard
**Files Reviewed:** 15

## Summary

Phase 04.1 ships substantial, well-structured security hardenings for the Excalidraw pipeline. The realpath walk-up validation, SVGO script/event-handler strip, dataURL scheme whitelist, and parsererror detection all address real exploit vectors documented in the Phase 4 security review. However, adversarial review surfaces one BLOCKER-class bug that **undercuts the escapeXml defense-in-depth claim** (`String.prototype.replace` special-pattern re-interpretation in the title/desc injection) plus several WARNING-level gaps in the new guards: incomplete regex coverage on the D-08 Helvetica replace, a narrow parsererror-detection bypass, and a dataURL scheme check that accepts arbitrary post-prefix garbage. The test suite is comprehensive for the happy-path and documented threat model but misses these boundary cases. The MDX/MD figure wraps are well-formed and bilingual; fixtures are deterministic and well-documented. Review is advisory.

## Critical Issues

### CR-01: `$&` / `$1` replacement-string injection in `<title>`/`<desc>` — escapeXml does NOT neutralize `$` metacharacters

**File:** `scripts/excalidraw-to-svg.mjs:233-236`
**Severity:** CRITICAL
**Category:** Bug / Security (output-integrity)

`String.prototype.replace(pattern, replacement)` interprets `$&`, `$1`..`$9`, `` $` ``, `$'`, and `$$` as special patterns when the replacement is a string. `escapeXml` only escapes `<`, `>`, `&`, `"`, `'` — it does NOT escape `$`. As a result, an author whose `meta.title` or `meta.descEn` contains literal `$&` (for example `"Diagram $& context"` or `"This costs $5"` adjacent to a digit) gets the SVG's entire opening `<svg ...>` tag (or arbitrary captured groups) re-injected into the `<title>` or `<desc>`. Demonstrated with:

```
meta.title = "Diagram $& post"
titleXml   = "Diagram $& post"              // escapeXml passes $ through
→ svg.replace(/<svg([^>]*)>/, `<svg$1><title>${titleXml}</title>...`)
→ <svg xmlns="..."><title>Diagram <svg xmlns="...">amp; post</title>...
```

This is the load-bearing failure mode the BL-02 latent-regex review finding was closed against (SEC-L02 closure note in 04.1-SUMMARY claims D-03's SVGO `removeScripts` makes the pipe "defense-in-depth … real, not claimed"). The `$&` vector is the analogous injection route through the replacement string — still present. Impact today:

- Breaks SVG document structure (nested `<svg>` inside `<title>`) — not XSS (SVG parsers treat nested tags as element content) but the SVG becomes visibly malformed.
- Later SVGO multipass may mis-parse the broken structure and produce unpredictable output.
- Claims about escapeXml fully neutralizing author-controlled strings are false until this is fixed.

No test covers this — `tests/unit/excalidraw-to-svg.test.ts` never exercises a `$`-containing title or desc.

**Fix:** Use the callback form of `String.prototype.replace` so the returned string is used verbatim without `$`-substitution:

```js
svgString = svgString.replace(
  /<svg([^>]*)>/,
  (_match, attrs) => `<svg${attrs}><title>${titleXml}</title><desc>${descXml}</desc>`
);
```

Or, alternatively, escape `$` in titleXml/descXml before the replace:

```js
const safe = (s) => s.replace(/\$/g, '$$$$');  // $$$$ → $$ in replacement → literal $
// then use `${safe(titleXml)}` in the template
```

Add a regression test:

```js
test('escapeXml :: dollar-replacement-chars — meta.title with $& does not inject svg attrs', () => {
  // write meta with title: "Cost $& $1 $2"
  // assert output <title> contains literal "Cost $& $1 $2" — no attribute re-injection
});
```

## Warnings

### WR-01: D-08 Helvetica replace misses common fallback-chain variants

**File:** `scripts/excalidraw-to-svg.mjs:254-257`
**Severity:** WARNING
**Category:** Bug / Robustness

The regex `/font-family="Helvetica(, Segoe UI Emoji)?"/g` only handles two forms:

1. `font-family="Helvetica"`
2. `font-family="Helvetica, Segoe UI Emoji"`

The following legitimate post-SVGO outputs are NOT rewritten:

- `font-family="Helvetica, Arial"` (user authored with explicit Arial, no Segoe)
- `font-family="Helvetica, sans-serif"` (user authored with generic fallback)
- `font-family='Helvetica'` (single-quoted — SVGO's normalisation normally keeps double but a future SVGO plugin or a hand-written SVG could produce single)
- `font-family="Helvetica Neue"` (different Apple font — wrong match, but relevant if Excalidraw ever changes its font picker)

Result on Windows: users who authored with `Helvetica, sans-serif` still get platform-default metrics (UX-04 regression not fully closed). The test `UX-04 :: helvetica-fallback-chain` only exercises the `Helvetica, Segoe UI Emoji` case — the others are untested and silently fail.

**Fix:** Widen the regex and add negative-assertion tests:

```js
const withFallback = optimized.replace(
  /font-family=(["'])Helvetica(?:,\s*[^"']+)?\1/g,
  'font-family="Helvetica, Arial, sans-serif"'
);
```

Then add tests for each variant (standalone, with Arial chain, with sans-serif chain, single-quoted).

### WR-02: Parsererror detection bypass — `<parsererror>` mid-document with no attributes escapes both regexes

**File:** `scripts/excalidraw-to-svg.mjs:218-227`
**Severity:** WARNING
**Category:** Security (detection completeness)

Two regexes guard against parsererror output:

- `/^<parsererror[\s>]/` — anchored at position 0, matches whitespace OR `>` after.
- `/<parsererror\s/` (applied to first 200 chars) — mid-document, matches only whitespace after.

The mid-document form **does not match `>`** after `parsererror`. A document like `<svg><parsererror>error message</parsererror></svg>` (no attributes, no whitespace) is NOT detected:

- `/^<parsererror[\s>]/` → false (doesn't start at index 0; starts with `<svg>`)
- `/<parsererror\s/` → false (`>` directly after `parsererror`, no `\s`)

In practice JSDOM always emits `<parsererror xmlns="http://www.mozilla.org/newlayout/xml/parsererror.xml">` (confirmed by reading `node_modules/jsdom/lib/jsdom/living/domparsing/DOMParser-impl.js:27`), so real bypass requires a future JSDOM change or a different XML parser stacked into the pipeline. But the filter has a correctness gap.

**Fix:** Align both regexes to match `[\s>]`:

```js
if (
  /^<parsererror[\s>]/.test(svgString) ||
  /<parsererror[\s>]/.test(svgString.slice(0, 200))
) {
  throw new Error(...);
}
```

Or collapse to a single unanchored regex that also covers the `/^<parsererror>/` case:

```js
if (/<parsererror[\s>]/.test(svgString.slice(0, 200))) { throw ... }
```

### WR-03: `ALLOWED_DATAURL_PREFIX` accepts arbitrary non-base64 garbage after the prefix

**File:** `scripts/excalidraw-to-svg.mjs:37, 138`
**Severity:** WARNING
**Category:** Security (input validation)

The regex `/^data:image\/(png|jpeg|gif);base64,/` only validates the scheme prefix. After the comma, any characters — including `<script>`, `\0`, `\r\n` injection, or completely non-base64 ASCII — pass the check. Example: `data:image/png;base64,<script>alert(1)</script>XXXXXXXXXXXXXXXXXXXX` is accepted.

Downstream behavior:

- The wrapper embeds this as `<image href="...">`. Browsers treat a broken base64 as a failed image load — no XSS, no script execution. SVGO's `removeScripts` further strips any script element that somehow got serialized.
- The 100 KB `FILES_BLOB_BUDGET` check uses `f.dataURL.length` (UTF-16 code units). Non-ASCII payloads compare differently against a byte-oriented budget; a 100 KB .length payload could be 300+ KB on-disk.

Neither is a live exploit, but the "whitelist" claim is weaker than stated. Phase 04.1 documentation asserts file://, http(s)://, and data:image/svg+xml are rejected — true. But the positive side does not validate payload well-formedness.

**Fix:** After prefix match, verify base64 alphabet:

```js
const DATAURL_PAYLOAD = /^[A-Za-z0-9+/]+=*$/;
if (!ALLOWED_DATAURL_PREFIX.test(f.dataURL)) { throw ... }
const payload = f.dataURL.slice(f.dataURL.indexOf(',') + 1);
if (!DATAURL_PAYLOAD.test(payload)) {
  throw new Error(`files[${id}].dataURL base64 payload contains non-base64 chars`);
}
```

Also use `Buffer.byteLength(f.dataURL, 'utf8')` instead of `f.dataURL.length` for the blob-budget accumulator, so the cap is enforced in actual bytes.

### WR-04: TOCTOU race between `validatePath` and `fs.writeFile`

**File:** `scripts/excalidraw-to-svg.mjs:109-130, 270-271`
**Severity:** WARNING
**Category:** Security (defense-in-depth hole)

`validatePath` walks up to the first existing ancestor, realpaths it, and reattaches the not-yet-created tail. This correctly rejects pre-existing symlinks that point outside the repo. But between `validatePath`'s return and the `fs.mkdir({ recursive: true })` + `fs.writeFile` at line 270-271, an attacker with write access to any resolved-but-non-existent ancestor can plant a symlink that points outside REPO_ROOT. The subsequent `fs.writeFile` follows it.

Concrete scenario:
1. Script validates `public/blog-assets/foo/diagram.svg` — `public/blog-assets/foo` does not exist yet; walks to `public/`, realpaths it (safe), reattaches tail (returns safe path, validation passes).
2. Between step 1 and `fs.mkdir(dirname)`, concurrent attacker with write access to `public/blog-assets/` creates symlink `public/blog-assets/foo` → `/etc`.
3. `fs.writeFile(path.join(REPO_ROOT, 'public/blog-assets/foo/diagram.svg'))` follows the symlink — writes to `/etc/diagram.svg`.

Real-world exploitability is low (attacker already has repo write, script usually runs as the developer). But the script is documented as symlink-hardened. The claim is weaker than it reads.

**Fix:** Open with `O_NOFOLLOW` (Node `fs.open` flag `'wx'` + a custom flag combination) or re-validate the parent directory's realpath at write time:

```js
// After mkdir, re-validate — parent now exists, realpath resolves current state.
await fs.mkdir(path.dirname(destPath), { recursive: true });
validatePath(destPath, 'destPath');  // second check — fails if symlink planted mid-race
await fs.writeFile(destPath, withFallback, 'utf8');
```

Alternatively, use `fs.openSync(destPath, fsConstants.O_WRONLY | fsConstants.O_CREAT | fsConstants.O_NOFOLLOW)` to fail-fast on a symlink destination.

### WR-05: `--quiet` flag filter strips `--quiet` anywhere in argv, including within file paths

**File:** `scripts/excalidraw-to-svg.mjs:157`
**Severity:** WARNING
**Category:** Bug / UX

`process.argv.slice(2).filter((a) => a !== '--quiet')` removes every token that exactly equals `--quiet`. A file named literally `--quiet` (unusual but valid on POSIX) would be silently dropped from positional args — the script would fail with "Usage: …" instead of a clear "cannot have --quiet as a filename" error.

A more important UX bug: the filter silently accepts unknown flags. `node script.mjs --verbose src.json dest.svg` treats `--verbose` as srcPath (which then fails the `.excalidraw.json` path check with a confusing error). No "unknown flag" warning.

Impact is minor (users don't typically name files `--quiet`), but the filter is over-aggressive and opaque.

**Fix:** Pre-parse flags into a set, reject unknown flag-shaped tokens, and validate positional count:

```js
const KNOWN_FLAGS = new Set(['--quiet']);
const rawArgs = process.argv.slice(2);
const positional = [];
const flags = new Set();
for (const arg of rawArgs) {
  if (arg.startsWith('--')) {
    if (!KNOWN_FLAGS.has(arg)) {
      console.error(`Unknown flag: ${arg}`);
      process.exit(1);
    }
    flags.add(arg);
  } else {
    positional.push(arg);
  }
}
const [srcPathArg, destPathArg] = positional;
const quiet = flags.has('--quiet');
```

This also eliminates the duplicate `process.argv.includes('--quiet')` at line 275 — a minor code-smell where `quiet` is re-derived from the original unmodified argv instead of the parsed `flags` set.

### IN-04: Duplicate quiet-flag derivation — inconsistent source of truth

**File:** `scripts/excalidraw-to-svg.mjs:157, 275`
**Severity:** INFO
**Category:** Code quality (duplication)

Line 157 filters `'--quiet'` out of argv to compute positional args. Line 275 re-checks the ORIGINAL `process.argv.includes('--quiet')`. If `process.argv` were ever mutated between these two lines (not currently, but future refactor risk), the derivations could disagree. Line 275 should read the filtered set or a parsed boolean computed once at the top:

**Fix:** Introduce a single `quiet` boolean at the top of `main()`:

```js
const rawArgs = process.argv.slice(2);
const quiet = rawArgs.includes('--quiet');
const argv = rawArgs.filter((a) => a !== '--quiet');
const [srcPathArg, destPathArg] = argv;
// … later …
if (!quiet) { /* success stdout */ }
```

Connects cleanly with WR-05's suggested unified flag parser.

### IN-06: `SVGO_CONFIG_MIRROR` test block duplicates production plugin config

**File:** `tests/unit/excalidraw-to-svg.test.ts:168-206`
**Severity:** INFO
**Category:** Code quality (duplication / drift risk)

The `Security :: strips-script-tags` test defines a local `SVGO_CONFIG_MIRROR` that mirrors `scripts/excalidraw-to-svg.mjs` `SVGO_CONFIG`. // NOTE: mirror omits the D-07 white-rect selector deliberately (scope: SEC-C02 only).

const SVGO_CONFIG_MIRROR = { /* ... */ };
